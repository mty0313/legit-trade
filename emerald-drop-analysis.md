# 绿宝石掉落实验分析

日期：2026-04-16

## 现象
有几次进入游戏后，发现地上有一个掉落的绿宝石。没有明确执行过“手动丢弃绿宝石”的动作，因此怀疑是否为 mod bug。

## 当前结论
当前代码库中，**没有“玩家进入游戏时主动生成并掉落绿宝石”** 的逻辑。

更可能的情况是：
1. 绿宝石曾经被放入交易 GUI 的输入槽；
2. 之后切换交易，或关闭 GUI；
3. 系统尝试把输入槽物品退回玩家背包；
4. 如果背包放不下，就会直接 `dropItem` 到地上。

因此，这更像是一个 **容器交互/输入槽设计问题**，而不是“进游戏时凭空刷出绿宝石”。

## 关键证据

### 1. 进游戏时只会同步配置，不会生成物品
`LegitTrade` 在玩家进入时只发送配置同步包：

- `src/main/java/com/trade/LegitTrade.java`
- `src/main/java/com/trade/network/ConfigSyncPacket.java`

同步内容只有：
- group
- input
- output
- inputCount
- outputCount
- xpReward

没有 `ItemStack` 创建，也没有 `dropItem(...)`。

### 2. 代码中明确会把物品丢地上的路径
在 `TradeScreenHandler` 中：

- `returnInputToPlayer(...)`
- `onClosed(...)`

其中 `returnInputToPlayer(...)` 的逻辑是：
- 先尝试放回玩家背包
- 如果背包塞不下，则 `player.dropItem(remaining, false)`

这说明：**某个真实存在于输入槽中的物品，确实可能在无“手动丢弃”动作的情况下被自动掉落到地上。**

### 3. 输入槽没有物品类型限制
`TradeScreenHandler` 的输入槽是普通 `Slot`，且 `quickMove(...)` 也允许把玩家背包中的任意物品塞进去。

这意味着：
- 绿宝石可以被误放入输入槽；
- 切换交易或关闭界面时，它会被系统尝试退回；
- 若无法退回，就会掉地上。

### 4. 配置中确实存在“产出绿宝石”的交易
当前配置里有：

- `minecraft:copper_ingot -> minecraft:emerald`

所以游戏过程中出现绿宝石本身并不奇怪；
但从代码上看，**“进入游戏时地上已有绿宝石”并不是登录同步直接产生的。**

## 最可能的解释
按概率排序：

1. **最高概率**：绿宝石曾被放进交易输入槽，之后在切换交易或关闭 GUI 时被自动丢出。
2. **次高概率**：玩家做过绿宝石相关交易，后续某次容器关闭流程把结果物品掉到地上。
3. **低概率**：进入游戏本身触发了绿宝石生成。当前代码分析基本不支持这一点。

## 这算不算 bug
不太像“刷物品 bug”或“无中生有的掉落 bug”。

更准确地说，这是一个 **UX / 交互设计缺陷**：
- 输入槽允许放任意物品；
- 自动切换交易/关闭界面时会自动退回；
- 背包装不下时直接掉地上；
- 玩家体感上会误以为“系统莫名其妙扔了一个物品出来”。

## 下次继续研究的方向
如果后续要继续排查/修复，建议重点看这几个方向：

1. 是否要限制输入槽只能放当前交易所需物品；
2. `quickMove(...)` 是否也应做同样限制；
3. 切换交易时，是否应避免自动 `dropItem`，改成更可控的返还策略；
4. 关闭 GUI 时，是否要对输入槽返还路径做更明确处理或加日志；
5. 如需复现，重点测试：
   - 往输入槽放入绿宝石；
   - 切换交易；
   - 关闭 GUI；
   - 背包留满格，观察是否稳定掉地。
